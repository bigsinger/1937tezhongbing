# 十二关任务恢复说明

## 结论

十二个正式关卡 `m000`—`m011` 的任务骨架不是只能在事后凭感觉重新编排。当前已经恢复并数据化：

- 关卡标题、任务简报图索引和主要目标；
- 目标之间的先后依赖；
- 救援、击毙、拾取、爆破、占点和撤离等条件类型；
- 多目标计数与按 scene 去重；
- 限时、角色损失和特殊失败条件；
- 关卡中的任务标记、爆破检测、出口、出生点和入口坐标；
- 全部必需目标完成后的胜利判定。

这些内容已经整理在 `game/data/missions.json`，由通用 `MissionState` 计算进度，并通过 `MissionRuntime` 与实际关卡实体连接。当前主场景已能从战斗和交互产生救援、角色击毙、物品拾取、爆破/占点、区域清敌、撤离、限时和必要角色损失事件。任务目标完成、失败和胜利会立即回写到界面。

这表示任务已经具备“世界动作 → 规范化事件 → 目标依赖/计数 → 胜负”的可玩工程闭环，不再只是静态任务清单；但它仍不是十二关逐段演出的完成版。对白、镜头、专属过场、触发半径、特定角色要求、增援节奏和若干候选 scene 仍需要逐关试玩并人工校准。

## 恢复证据链

### 1. VWF/SLIST 关卡事实

十二个正式 VWF 均可解析到精确文件末尾，共有 19,199 条实体记录。每条实体可关联到 DBL 对象名称、类别、世界坐标、参考坐标和可选巡逻数据。

任务相关 DBL 项已确认：

| DBL ID | 原显示名 | 规范化类型 | 用途 |
|---:|---|---|---|
| 1001 | 藏尸处 | `corpse_state` | 尸体/发现状态锚点 |
| 1008 | 检测视线精灵 | `sight_detector` | 视线或剧情检测 |
| 1010 | 进入表示 | `entrance` | 进入点 |
| 1011 | 日军出兵处 | `enemy_spawn` | 敌人出生点 |
| 1018 | 标注 | `marker` | 编辑器中的可见任务标记 |
| 1019 | 检测爆炸精灵 | `explosion_detector` | 爆破目标检测 |
| 1020 | 检测出口精灵 | `exit_detector` | 撤离/占点检测 |

其中 24 个爆破检测和 12 个出口检测都能在 32 像素内唯一配对到一个任务标记。另有 7 个独立任务标记，用于军服箱、剧情 NPC 等叙事位置。位置和配对关系可以机械生成并验证。

### 2. 任务控制流

VWF/SLIST 本身保存场景事实，不保存一份完整的高层任务图。已审计 SAV 也按其结构精确结束，没有附加任务脚本；`M1937.SI0` 中可识别的 IBLOCK 是 320×240 存档缩略图。

目标依赖、计数、限时和胜负分支来自原程序中十二关任务控制流的恢复。简报文本和任务图用于给控制流中的实体指针、计数器和分支赋予可读语义。恢复结果再与 VWF 中实际存在的 scene 和锚点数量交叉验证。

### 3. 新运行时数据模型

规范化任务定义不复制旧引擎代码，而是使用事件条件：

```json
{
  "id": "destroy_airfield",
  "depends_on": [],
  "condition": {
    "event": "trigger_activated",
    "where": {"display_name": "检测爆炸精灵"},
    "required_count": 6,
    "unique_by": "scene_index"
  }
}
```

`MissionState` 对事件名称和 payload 做通用匹配，因此同一个状态机可以处理十二关，而不需要为每关复制一套胜负代码。新增的 `MissionRuntime` 位于世界系统和状态机之间，负责：

- 在关卡开始时确认每个 `scene_bindings` 都存在于当前 `level.json`，并核对爆破/出口等绑定的锚点类型；
- 拒绝缺少 scene 引用、跨关卡或未绑定 scene 的救援、拾取、击毙、剧情锚点、爆破和撤离事件；
- 把救援、拾取、角色击毙、剧情锚点和区域清敌等事实按稳定身份去重；
- 暂存依赖尚未满足的持久事实，在前置目标完成后重新回放，避免“先拾物、后完成击毙”造成永久卡关；
- 保持撤离区域为瞬时判定，不把一次短暂进入错误缓存为永久事实；
- 只允许 `advance_time()` 产生限时失败，防止世界系统伪造 `time_expired`。

当前世界事件入口为：

| 世界行为 | 规范化事件 |
|---|---|
| 靠近并按 `E` 解救任务 NPC | `entity_rescued` |
| 击毙与任务角色绑定的实体 | `role_eliminated` |
| 按 `E` 拾取场景物品或角色掉落 | `item_acquired` |
| 按 `E` 激活爆破点/制高点 | `trigger_activated` |
| m008 按 `F` 手动引爆矿坑炸药 | `explosion`（四处布置完成前触发失败） |
| 到达剧情锚点 | `story_anchor_reached` |
| 清除当前关卡敌人（现用于 m009 可选目标） | `area_hostiles_cleared` |
| 存活队员与已救 NPC 到达出口 | `trigger_activated` / `party_at_trigger` |
| 必要队员或护送 NPC 死亡 | `required_character_lost` |
| 计时达到关卡上限 | `time_expired` |

第一关 `m000` 的两个救援对象（scene 1427、1428）和出口（scene 1600）已按关卡实体绑定：营救两人后，存活队员和获救人员都进入 128 像素出口范围才完成撤离；必要角色死亡会失败。这是当前最明确的端到端任务闭环，也是其余关卡复用事件协议的基线。

## 已识别的数据差异

恢复过程保留了不一致信息，避免用一个看似整齐的名称掩盖证据：

- `m005`：简报目标叫“阿贵”，scene 736 的 DBL 显示名却是“阿全”；任务数据使用稳定角色 ID `m005_agui`，并记录这项差异；
- `m004`：携带计划书的军官暂绑定候选山本 scene 2637；原资源中另有候选 scene 2548，当前绑定可形成可玩掉落闭环，但不能在运行对照前称为最终考据结论；
- `m006`：剧情中的加藤孝一由旧引擎 type 22 / DBL“王老板”资源承担；任务规则使用剧情角色 ID，而不是把资源显示名当作身份；
- `m009`：原控制流中明确可验证的必要计数是四个列车爆破目标；简报中的交接文件和清剿车站先作为可选目标保留，等待实际运行对照；
- `m011`：原版存在爆破目标指针覆盖缺陷，复刻默认按简报修正，见下一节。

这些项目属于需要人工判读和运行校验的“语义层”，但不会阻止场景坐标、锚点计数和任务状态骨架自动化。

## m011 原版指针覆盖缺陷

第十二关有六个爆破检测实体：scene 1348、1349、1350、1351、1352、1353。任务简报也要求炸毁军机、指挥塔和油库等共六处目标。

原版初始化逻辑却把六个目标指针连续写入同一个存储位置。前五次赋值都被下一次覆盖，最终只剩 scene 1353；随后胜利检查实际上只观察这个最后目标。这是原版实现缺陷，不是关卡中缺少五个锚点。

复刻任务数据的默认意图是：

```text
required_count        = 6
unique_by             = scene_index
intended scene binding = 1348..1353
```

也就是要求六个不同爆破目标，和简报及关卡布置一致。`MissionRuntime` 现在会把事件限制在当前关卡 `scene_bindings`，并确认六个 scene 都是实际爆破锚点；`unique_by = scene_index` 防止同一目标重复计数。未来如果确有保存原版异常行为的需求，可以另加显式的 `strict-original` 兼容选项；当前没有实现这个开关。

## 哪些内容可以自动化

在格式和语义映射已经确认之后，下列工作可以由工具生成或严格校验：

- 十二关地图尺寸、19,199 个实体和全部坐标；
- DBL 资源/显示名/类别关联；
- 巡逻当前航点索引、缓存航点世界坐标与 waypoint；其中持久标志 `+0x0C` 的启停语义尚未得到运行时代码证明；
- 七类任务锚点的 scene、坐标和类型，以及标记/爆破/出口/出生/入口五类锚点的逐关数量清单；
- 36 个爆破/出口检测与标记的一对一近邻配对；
- 十二关任务 JSON 的 schema、目标 ID、依赖、计数、去重字段和限时；
- 运行时目标进度、失败和胜利判定。
- 任务 scene 白名单、锚点类型、事件去重、依赖回放和瞬时出口语义校验。

这使后续关卡实现以“连接事件”为主，而不是重新猜测每个爆破点和出口在哪里。

## 哪些内容仍需人工校准

静态数据和控制流不能完整给出以下体验层信息：

- 对白文本出现时机、人物语气和字幕停留时间；
- 镜头移动、过场站位、动作衔接和音效同步；
- 检测区域半径、是否要求特定角色、多人同时到达的判定细节；
- 敌人 AI 配合、增援时机、警戒反应和难度数值；
- m004 计划书军官候选 scene 2637/2548 的原版运行对照；
- m008 已提供 `F` 手动引爆并接通“布置四处炸药前引爆则失败”的事件分支；仍缺少原版引信、爆炸传播、延时和演出对象；
- m009“清剿车站”当前以全关敌人清零作为可玩近似，尚未恢复车站区域边界；
- m010 制高点当前采用接近锚点后按 `E` 的通用交互，尚未恢复占领持续时间、敌人控制权等细节；
- m005/m006/m009 等名字或简报差异的最终呈现；
- 原版缺陷是修正还是提供兼容模式的产品决策。

因此最合理的流程是：先由恢复数据生成任务骨架和自动测试，再逐关试玩、录制和校准演出，而不是先手工搭十二套脚本再反向猜测原作。

当前十二关共用同一事件协议和通用交互器；`m000` 已形成明确的营救—护送—撤离胜负闭环，任务角色击毙/掉落、爆破/占点和限时也已接线。下一阶段不是再造一套任务状态机，而是逐关核对候选角色、区域边界、特定队员条件、爆炸时序、对白和演出，并为每关增加可重复的端到端回放。
