# 十二关任务恢复说明

## 结论

十二个正式关卡 `m000`—`m011` 的任务骨架不是只能在事后凭感觉重新编排。当前已经恢复并数据化：

- 关卡标题、任务简报图索引和主要目标；
- 目标之间的先后依赖；
- 救援、击毙、拾取、爆破、占点和撤离等条件类型；
- 多目标计数与按 scene 去重；
- 限时、角色损失和特殊失败条件；
- 关卡中的任务标记、爆破检测、出口、出生点和入口坐标；
- 全部必需目标完成后的胜利判定。

这些内容已经整理在 `game/data/missions.json`，由通用 `MissionState` 计算进度，并通过 `MissionRuntime` 与实际关卡实体连接。当前主场景已能从战斗和交互产生救援、角色击毙、物品拾取、爆破/占点、区域清敌、撤离、限时和必要角色损失事件。任务目标完成、失败和胜利会立即回写到界面。

这表示任务已经具备“世界动作 → 规范化事件 → 目标依赖/计数 → 胜负”的可玩工程闭环，不再只是静态任务清单；但它仍不是十二关逐段演出的完成版。m004、m009、m010 的关键控制流语义已经定案；对白、镜头、专属过场、增援节奏、AI 配合和难度仍需要逐关试玩并人工编排。

## 恢复证据链

### 1. VWF/SLIST 关卡事实

十二个正式 VWF 均可解析到精确文件末尾，共有 19,199 条实体记录。每条实体可关联到 DBL 对象名称、类别、世界坐标、参考坐标和可选巡逻数据。

任务相关 DBL 项已确认：

| DBL ID | 原显示名 | 规范化类型 | 用途 |
|---:|---|---|---|
| 1001 | 藏尸处 | `corpse_state` | 尸体/发现状态锚点 |
| 1008 | 检测视线精灵 | `sight_detector` | 视线或剧情检测 |
| 1010 | 进入表示 | `entrance` | 进入点 |
| 1011 | 日军出兵处 | `enemy_spawn` | 敌人出生点 |
| 1018 | 标注 | `marker` | 编辑器中的可见任务标记 |
| 1019 | 检测爆炸精灵 | `explosion_detector` | 爆破目标检测 |
| 1020 | 检测出口精灵 | `exit_detector` | 撤离/占点检测 |

其中 24 个爆破检测和 12 个出口检测都能在 32 像素内唯一配对到一个任务标记。另有 7 个独立任务标记，用于军服箱、剧情 NPC 等叙事位置。位置和配对关系可以机械生成并验证。

### 2. 任务控制流

VWF/SLIST 本身保存场景事实，不保存一份完整的高层任务图。已审计 SAV 也按其结构精确结束，没有附加任务脚本；`M1937.SI0` 中可识别的 IBLOCK 是 320×240 存档缩略图。

目标依赖、计数、限时和胜负分支来自原程序中十二关任务控制流的恢复。简报文本和任务图用于给控制流中的实体指针、计数器和分支赋予可读语义。恢复结果再与 VWF 中实际存在的 scene 和锚点数量交叉验证。

### 3. 新运行时数据模型

规范化任务定义不复制旧引擎代码，而是使用事件条件：

```json
{
  "id": "destroy_airfield",
  "depends_on": [],
  "condition": {
    "event": "trigger_activated",
    "where": {"display_name": "检测爆炸精灵"},
    "required_count": 6,
    "unique_by": "scene_index"
  }
}
```

`MissionState` 对事件名称和 payload 做通用匹配，因此同一个状态机可以处理十二关，而不需要为每关复制一套胜负代码。新增的 `MissionRuntime` 位于世界系统和状态机之间，负责：

- 在关卡开始时确认每个 `scene_bindings` 都存在于当前 `level.json`，并核对爆破/出口等绑定的锚点类型；
- 拒绝缺少 scene 引用、跨关卡或未绑定 scene 的救援、拾取、击毙、剧情锚点、爆破和撤离事件；
- 把救援、拾取、角色击毙、剧情锚点和区域清敌等事实按稳定身份去重；
- 暂存依赖尚未满足的持久事实，在前置目标完成后重新回放，避免“先拾物、后完成击毙”造成永久卡关；
- 保持撤离区域为瞬时判定，不把一次短暂进入错误缓存为永久事实；
- 只允许 `advance_time()` 产生限时失败，防止世界系统伪造 `time_expired`。

当前世界事件入口为：

| 世界行为 | 规范化事件 |
|---|---|
| 靠近并按 `E` 解救任务 NPC | `entity_rescued` |
| 击毙与任务角色绑定的实体 | `role_eliminated` |
| 按 `E` 拾取场景物品或角色掉落 | `item_acquired` |
| 按 `E` 激活爆破点等持久任务锚点（m010 制高点除外） | `trigger_activated` |
| m008 按 `F` 手动引爆矿坑炸药 | `explosion`（四处布置完成前触发失败） |
| 到达剧情锚点 | `story_anchor_reached` |
| 清除当前关卡全部 faction 1 敌人（m009 修复后必需目标） | `area_hostiles_cleared` |
| m010 四名指定队员分别同时进入四个制高点 | `simultaneous_zones_occupied` |
| 存活队员与已救 NPC 到达出口 | `trigger_activated` / `party_at_trigger` |
| 必要队员或护送 NPC 死亡 | `required_character_lost` |
| 计时达到关卡上限 | `time_expired` |

第一关 `m000` 的两个救援对象（scene 1427、1428）和出口（scene 1600）已按关卡实体绑定：营救两人后，存活队员和获救人员都进入 128 像素出口范围才完成撤离；必要角色死亡会失败。这是当前最明确的端到端任务闭环，也是其余关卡复用事件协议的基线。

### 4. 爆破目标与背包炸药策略

真实 DBL/VWF 可以证明每关 DBL 998 炸药拾取物和爆破 scene 的数量，但尚不能证明原版每个目标的消耗调用。因此当前 `charge_policy` 是基于恢复地图物资闭环的明确重制策略，来源状态为 `remake_policy_from_recovered_map_inventory`：

| 关卡 | DBL 998 / 爆破目标 | 模式 | 交互语义 |
|---|---:|---|---|
| m001 | 1 / 2 | `preplanted` | 两个兵营目标不扣背包 |
| m002 | 1 / 1 | `inventory_required` | 每个目标需要并消耗 1 份炸药 |
| m003 | 6 / 5 | `inventory_required` | 五个目标各消耗 1 份，保留 1 份余量 |
| m004 | 0 / 2 | `preplanted` | 两座粮仓按预置引燃物处理，不扣背包 |
| m008 | 4 / 4 | `inventory_required` | 四个工事目标各消耗 1 份 |
| m009 | 9 / 4 | `inventory_required` | 四个列车目标各消耗 1 份 |
| m011 | 4 / 6 | `preplanted` | 按简报修复为六目标后不强行制造两份炸药，不扣背包 |

`MissionData` 会拒绝未知模式、目标数量与 scene 绑定不一致，以及 `inventory_required` 但地图炸药少于所需数量的任务。运行时先检查背包，再发布 `trigger_activated`；只有事件通过当前关卡/scene 白名单且发布成功后才扣除，失败或重复交互均不会消耗。没有 `charge_policy` 的非爆破关保守回退为不消费的预置锚点，绝不会因为背包中偶然存在炸药而误扣。

### 5. 任务事件媒体提示

任务可以可选声明 `media_cues`，把 `audio`、`dialogue`、`movie` 或 `ending` 绑定到四类已验证时机：

| 段 | 键 | 触发条件 |
|---|---|---|
| `on_start` | 单个 cue | 非 headless 简报关闭后 |
| `on_objective` | objective ID | 该目标首次完成时 |
| `on_story_anchor` | role ID | 剧情锚点事件真正完成新目标时 |
| `on_victory` | 单个 cue | 任务进入胜利状态时 |

每个 cue 必须把 `source_status` 标为 `recovered_media_mapping`、`remake_editorial` 或 `mixed`；未知标签、引用不存在的目标/锚点，或缺少相应媒体字段都会让任务目录校验失败。当前实际接线只有：

- m000：关闭简报后显示明确标为 `remake_editorial` 的教程提示；首次完成 `rescue_pengxin` 后播放老赵 `acknowledge`，其触发编排同样标为重制编辑；
- m006：scene 1461 的接头锚点首次完成 `follow_contact` 时显示 `remake_editorial` 提示对白；重复发布同一持久事实不会重播模态对白；
- m011：胜利时按 `recovered_media_mapping` 打开与原资源映射一致的结局图，并在资源缺失时显示“十二关任务全部完成”。

这证明任务事件与媒体导演之间已经有数据驱动接口，也避免把编辑提示伪装成原剧情；它不表示十二关对白、镜头或过场已经恢复完成。

## m004、m009、m010 最终语义

### m004：计划书携带者

原任务 case 5 检查物品 101；关卡 VWF 的 `auxiliary[0]` 携带物记录显示 scene 2637 持有该物品，同名候选 scene 2548 不持有。因此 `m004_plan_officer = [2637]` 已从候选升级为最终绑定。击毙 scene 2637 后生成带 `original_item_id = 101` 的计划书掉落，再由拾取事件完成后继目标。

### m009：全关清敌是修复后的设计意图

原 case 10 会扫描全关仍存活的 faction 1 单位，而不是检查一个可恢复的“车站几何边界”；但该扫描结果因控制流缺陷没有参与最终判断。原版实际必要路径近似只保留四个列车爆破目标和老赵存活。复刻默认按简报与这段全场扫描修复设计意图，要求：

1. 从 scene 1610、1611 取得两份物品 101 交接文件；
2. 全关 faction 1 日军清零；
3. 激活 scene 1714—1717 四个列车爆破目标。

所以当前“全关清零”不是尚未找到车站边界时的临时近似，而是对原控制流未使用结果的明确修复决定。未来若提供 `strict-original` 模式，可只复现原版实际生效路径；该开关尚未实现。

### m010：四人同时占点

原 evaluator 对四个出口检测 scene 1620—1623 做连续实时检查。每个区域使用包含边界的 128 世界像素半径，必须能给四个区域分别分配不同的存活队员，且候选人恰好是老赵、强子、大牛、古明；铁蛋不能替代。四人必须在同一次评估中同时满足，不累计先前占领，不要求停留时间，也不要求先清敌。主场景当前每 0.2 秒自动检查，因此不再通过靠近后按 `E` 逐点激活。任务仍受 3,600 秒时限和这四名角色的存活约束。

## 已识别的数据差异

恢复过程保留了不一致信息，避免用一个看似整齐的名称掩盖证据：

- `m005`：简报目标叫“阿贵”，scene 736 的 DBL 显示名却是“阿全”；任务数据使用稳定角色 ID `m005_agui`，并记录这项差异；
- `m004`：计划书军官已确认是 scene 2637；scene 2548 的同名资源不携带原任务检查的物品 101，不再列为待定候选；
- `m006`：剧情中的加藤孝一由旧引擎 type 22 / DBL“王老板”资源承担；任务规则使用剧情角色 ID，而不是把资源显示名当作身份；
- `m009`：原控制流确有全关 faction 1 存活扫描，但结果因缺陷未被使用；复刻默认按简报修复为两份文件、全关清敌和四个列车爆破均为必需目标；
- `m011`：原版存在爆破目标指针覆盖缺陷，复刻默认按简报修正，见下一节。

这些项目属于需要人工判读和运行校验的“语义层”，但不会阻止场景坐标、锚点计数和任务状态骨架自动化。

## m011 原版指针覆盖缺陷

第十二关有六个爆破检测实体：scene 1348、1349、1350、1351、1352、1353。任务简报也要求炸毁军机、指挥塔和油库等共六处目标。

原版初始化逻辑却把六个目标指针连续写入同一个存储位置。前五次赋值都被下一次覆盖，最终只剩 scene 1353；随后胜利检查实际上只观察这个最后目标。这是原版实现缺陷，不是关卡中缺少五个锚点。

复刻任务数据的默认意图是：

```text
required_count        = 6
unique_by             = scene_index
intended scene binding = 1348..1353
```

也就是要求六个不同爆破目标，和简报及关卡布置一致。`MissionRuntime` 现在会把事件限制在当前关卡 `scene_bindings`，并确认六个 scene 都是实际爆破锚点；`unique_by = scene_index` 防止同一目标重复计数。未来如果确有保存原版异常行为的需求，可以另加显式的 `strict-original` 兼容选项；当前没有实现这个开关。

## 哪些内容可以自动化

在格式和语义映射已经确认之后，下列工作可以由工具生成或严格校验：

- 十二关地图尺寸、19,199 个实体和全部坐标；
- DBL 资源/显示名/类别关联；
- 巡逻当前航点索引、缓存航点世界坐标与 waypoint；其中持久标志 `+0x0C` 的启停语义尚未得到运行时代码证明；
- 七类任务锚点的 scene、坐标和类型，以及标记/爆破/出口/出生/入口五类锚点的逐关数量清单；
- 36 个爆破/出口检测与标记的一对一近邻配对；
- 十二关任务 JSON 的 schema、目标 ID、依赖、计数、去重字段和限时；
- 运行时目标进度、失败和胜利判定。
- 任务 scene 白名单、锚点类型、事件去重、依赖回放和瞬时出口语义校验。

这使后续关卡实现以“连接事件”为主，而不是重新猜测每个爆破点和出口在哪里。

## 哪些内容仍需人工校准

静态数据和控制流不能完整给出以下体验层信息：

- 对白文本出现时机、人物语气和字幕停留时间；
- 镜头移动、过场站位、动作衔接和音效同步；
- 除 m010 已恢复的 128 像素/四名指定角色/同时占点外，其余任务的检测区域、阶段性特定角色和多人条件；
- 敌人 AI 配合、增援时机、警戒反应和难度数值；
- m008 已提供 `F` 手动引爆并接通“布置四处炸药前引爆则失败”的事件分支；仍缺少原版引信、爆炸传播、延时和演出对象；
- m005/m006 等名字或简报差异的最终呈现；
- 原版缺陷是修正还是提供兼容模式的产品决策。

因此最合理的流程是：先由恢复数据生成任务骨架和自动测试，再逐关试玩、录制和校准演出，而不是先手工搭十二套脚本再反向猜测原作。
