# 导航、动态占位、敌人感知与攻击

本文只把已经由资源、反汇编调用路径或真实关卡统计交叉验证的结论称为“原版参数”。复刻已经接入的行为和仍未完成的完整玩法分别列出，避免把几何核心误写成整套战斗已经完成。

## 1. 五张地图层各司其职

| 层 | 原版语义 | 复刻中的处理 |
|---:|---|---|
| L1 | 地块索引 | 高 16 位 tile-group ID、低 16 位 tile index，经 TLG 图集重建可见的 `terrain.png` |
| L2 | 视线/射击遮挡 | 普通敌人视线和所有已确认攻击的格线遮挡 |
| L3 | 移动障碍 | 玩家、敌人八方向 A* 和运行时动态占位的权威平面 |
| L4 | 事件设定 | 十二个正式关卡全为零，不据此虚构任务触发器 |
| L5 | 手动移动障碍修正 | 原样导出供研究；不与 L3 做运行时 OR |

L2 与 L3 不能合并：有些位置可看/可射但不能走，也可能能走却遮挡视线。把 L5 当作第二张碰撞层会制造原版不存在的封路和卡角。

L2/L3 已验证的值约定为：

```text
0                    开放
1                    静态障碍
scene_index + 1000   对应 SLIST 场景实体的占用
```

## 2. 静态地图与动态实体

`NavigationGridData` 永久保留导入的 L2/L3 快照，不在角色移动时改写源数组。`DynamicOccupancyGrid` 只接管明确注册为可移动的玩家和敌人；建筑、树木、车辆等未注册 scene 仍按源层阻挡，不能因为值大于 1000 就被全局放行。

动态占位遵守原版的所有权规则：

- 注册角色时以 VWF `ReferenceX/Y` 对应的源格为足印锚点，只取得包含该锚点的八连通分量；这避免把同一 scene 的历史/分离标记误合并成跨地图的巨型碰撞体，找不到足印才回退为单格；
- 寻路只忽略寻路者自己，其他角色、静态值 1 和未注册 scene 继续阻挡；
- 多格足印寻路会为每个候选锚点检查整副足印，不会只让锚点绕墙而让身体穿墙；
- 搬移先检查整段和全部新足印，任一冲突即整体失败；
- 提交时只移除 owner 等于自己的旧格，不会擦除墙或其他实体；
- 角色当前位置、L2 动态遮挡和终点预留一起更新，出生点不留“空气墙”；
- 4 像素密集采样防止大 `delta` 穿墙，12 像素角色间距、同帧移动边预留和对角交叉检查防止互穿；
- 玩家被阻挡持续 0.25 秒后按当前动态图重新寻路；敌人采用 0.65—1.00 秒的确定性错峰退避，避免拥挤处同时重算，不会退化为穿墙直线移动。

所有角色共享同一 overlay。m000 真实运行回归同时注册 1 名可控队员和 54 名敌人，巡逻过程中源 L2/L3 逐值不变。

## 3. 自动寻路与视线

`M37NAV1` 中间格式保留 L2—L5 的原始 `uint32`：

```text
char[8] magic = "M37NAV1\0"
uint32  version = 1
uint32  width, height, cell_width, cell_height
uint32  layer_count = 4
重复四次：uint32 layer_id（2、3、4、5）+ uint32 values[width*height]
```

Godot 使用 `AStarGrid2D` 的八方向模式 `DIAGONAL_MODE_ONLY_IF_NO_OBSTACLES`，因此不会从两个贴角障碍间斜穿。目标在障碍中时会寻找附近可走格；不可达时只返回同侧 partial path，不会补一条穿墙直线。

L2 采用 supercover 整数格线，沿途格和对角经过的两个侧格都会检查。普通敌人视线和手枪、步枪、机枪、投掷及近战攻击都受 L2 阻挡；观察者和目标可以忽略各自动态占位，但第三方动态实体仍会遮挡。

## 4. 原版敌人感知参数

DBL sprite header 的 h8（原记录 `+548`）是阵营：1 敌方、2 中立、3 友方；h12（`+564`）是特殊感知标记。十二关角色统计为敌方 656、中立 85、友方 31，另有 13 个 faction 0 角色类记录。特殊标记只出现在 DBL 1007 的军犬，正式关卡共 5 只。

普通敌人的感知区域是等距投影椭圆：横半径 640、纵半径 320 像素，二者都是 20 个 32×16 网格单元。角色方向值 1—8 对应的屏幕投影中心角和瞬时半角为：

```text
中心角：[45, 90, 135, 180, 225, 270, 315, 0]
半角：  [30, 45,  60,  45,  30,  45,  60, 45]
```

原传感器以 3° 步进扫描逻辑上的 90° 扇区；等距投影使屏幕角半宽随方向变化。感知椭圆分近、远两半：匍匐目标只在远半区规避普通识别，进入近半区仍会暴露。普通感知还必须通过 L2 视线。

军犬使用 320×160 像素、360° 的特殊感知路径，并绕过普通 L2 视线条件。代码只称它为“特殊感知”，不把未被程序证实的嗅觉解释写成事实。

## 5. 原版攻击范围

攻击距离同样使用等距椭圆，纵半径为横半径的一半；所有下列已确认攻击都检查 L2：

| type | 动作 | 横/纵半径（像素） |
|---:|---|---:|
| 1 | 手枪（action 6） | 480 / 240 |
| 2 | 步枪（action 9） | 700 / 350 |
| 3 | 机枪（action 10） | 576 / 288 |
| 4 | 匕首（action 13） | 32 / 16 |
| 5 | 大刀（action 12） | 32 / 16 |
| 6 | 飞镖（action 14） | 320 / 160 |
| 7 | 弹弓（action 15） | 320 / 160 |

## 6. 已接入的敌人行为与边界

敌人已从静态预览升级为运行时角色，并与玩家共用动态占位：

- 读取阵营、出生方向、生命值、默认攻击和军犬特殊感知字段；
- 把巡逻网格点换算为 `x*32+16, y*16+8`，按 L3 A* 循环巡逻；
- 跳过已经到达的连续巡逻点；整条路线退化为当前位置时进入低频等待，避免每个物理帧重复请求 A*；
- 感知玩家后按 L2、方向扇区和近/远区进入追击；
- 进入对应武器椭圆且 L2 可达时进入攻击态；
- 丢失视线后前往最后位置搜索，超时再恢复巡逻；
- 攻击复核采用原版确认的 20—39 基础 tick 区间，并使用确定性的 scene 序列。

巡逻对象字段已经纠正：旧名 `Behavior` 实际是当前航点索引，旧名 `OriginX/Y` 实际是该索引所指航点的缓存世界坐标。十二关有 780 个巡逻对象，但只有 516 条非空路线；264 个空对象不能当作实际巡逻。`persistent_flag` 十二关全为 1，但其“启停”语义尚未证实，复刻只以路线是否非空决定能否巡逻。

当前攻击态会发出带 attack type 的战斗事件，但伤害、弹药、命中帧、受伤/死亡动作、尸体发现、警报传播和任务击杀事件尚未全部接入，因此还不能宣称完整战斗复刻完成。

## 7. 自动回归

- .NET 合成格式测试：154 项；
- Godot 逻辑测试：123 项；
- 十二关真实资源、导航、阵营、攻击、生命值、军犬、巡逻和动画：6,631 项；
- m004 高密度导航压力回归固定采样 120 个物理帧，并同时断言敌人确实移动、A* 请求量处于 20—500 次且总寻路耗时不超过 2 秒；此前退化巡逻点造成的 28,783 次重复请求已由测试上限防回归；
- m000 窗口渲染回归：54 名敌人、43 条敌方巡逻路线，采样期间 35 名敌人实际移动；360 帧平均 16.667 ms、0 帧超过 20 ms、最大 18.805 ms（GTX 1050 Ti，Compatibility renderer，60 FPS 限制）。

导航或 AI 修改至少要继续验证：禁止斜穿墙角、源层不变、动态占位原子迁移、第三方遮挡、多人目标不重叠、敌人巡逻无出生点 ghost，以及十二关都能加载。
